Mitos del software — todos FALSOS
  • El hardware es más importante que el software
  • El software es fácil de desarrollar
  • Software = solo programas ejecutables
  • Desarrollar software = solo programar
  • Es natural que el software contenga errores

Teoría

Conceptos clave para las preguntas cortas (2,5 pts)

Ciclos de vida del softwareMuy preguntado
▼

Cascada

Fases secuenciales: Análisis→Diseño→Codificación→Integración→Mantenimiento. Terminas cada fase antes de empezar la siguiente.

Ciclo en V

Como cascada pero jerarquizado. Izquierda: descomposición. Derecha: integración. Diferencia verificación de validación.

Prototipo rápido

"Usar y tirar". Solo para ganar experiencia en análisis/diseño. Se desarrolla rápido.

Prototipo evolutivo

Iterativo sobre cascada. Cada iteración produce una versión más completa del sistema final.

Espiral

4 cuadrantes: Planificación → Análisis de riesgo → Ingeniería → Evaluación. Guiado por riesgo.

XP

4 valores: Sencillez, Comunicación, Retroalimentación, Valentía. Ciclos cortos, pruebas continuas.

Cascada vs V: el modelo en V añade jerarquía y diferencia verificación (¿cumple especificación?) de validación (¿cumple necesidades del usuario?).
Acoplamiento y cohesiónMuy preguntado
▼

Objetivo: acoplamiento débil + cohesión alta

Un módulo bueno hace una sola cosa bien definida (cohesión alta) y depende lo mínimo de otros módulos (acoplamiento débil).
Abstracción, herencia y genericidadExamen 2013
▼

Abstracción

Identificar elementos significativos ocultando detalles irrelevantes. Tipos: funcional (funciones), TAD (tipo abstracto de datos), máquina abstracta.

Herencia

Los hijos heredan estructura y operaciones del padre. Permiten reutilizar y extender. Ej: Vehículo → Coche → Coche eléctrico.

Genericidad

Elemento genérico válido para varios tipos. En C++: templates. Ej: Lista<T> sirve para Lista<int>, Lista<String>, etc.

Diferencia clave: abstracción define QUÉ hace un módulo ocultando el CÓMO. Herencia permite reutilizar y extender lo ya definido.
Pruebas de softwareExamen jun 2014
▼

Caja negra

Ignora estructura interna. Solo verifica entrada-salida. La puede aplicar el cliente. Métodos: clases de equivalencia, valores límite.

Caja transparente

Conoce la estructura interna. Cubre todos los caminos del código. Métodos: cubrimiento lógico, pruebas de bucles.

Big Bang

Integra todo de golpe. Solo válido para sistemas pequeños.

Descendente

Parte del módulo principal + sustitutos. Ve las posibilidades desde el principio.

Ascendente

Parte de módulos hoja + gestores. Facilita trabajo paralelo.

Validación

Prueba que el sistema cumple los requisitos del cliente. Puede revelar mensajes ininteligibles, UI mal diseñada, etc.

Una prueba tiene éxito si descubre un error. Si no aparecen errores, NO se garantiza la calidad.

▼
SRD
Documento de Requisitos de Software. Producto del análisis. Especifica el QUÉ debe hacer el sistema.
SDD / ADD
Documento de Diseño. El CÓMO lo hace. Arquitectónico (ADD) + Detallado (DDD).
Código fuente
Producto de la codificación. Debidamente comentado.
SQAP
Plan de Garantía de Calidad. Cubre organización, ciclo de vida, revisiones, pruebas y mantenimiento.
CMMI
5 niveles: 0 Inexistente → 1 Inicial → 2 Repetible → 3 Definido → 4 Gestionado → 5 Optimizado.

Truco composición vs agregación: si B no puede existir sin A → composición ◆. Si B puede existir independientemente → agregación ◇.
Ejercicio: MonopolyExamen sept. 2014
▼

40 casillas, 28 propiedades en grupos, hasta 6 jugadores, dinero, casas y hoteles.

Clave: Casilla → Propiedad es herencia porque "Propiedad es un tipo de Casilla". Propiedad → Edificio sería composición (los edificios no existen sin la propiedad).
Ejercicio: Partículas elementalesExamen jun. 2012
▼

Leptones (spin 1/2·h, sin interacción fuerte), Hadrones → Mesones (spin entero) y Bariones (spin semientero).

Clave: el atributo spin va en Partícula porque todas lo tienen. Cada subclase restringe su valor permitido.

Modelo E-R

Entidad-Relación + tablas relacionales

Símbolos del diagrama E-R
Traducción a tablas SQL
1 : 1
Se pueden fundir las dos entidades en una sola tabla.
1 : N
Clave primaria del "1" va como clave foránea en la tabla del "N".
N : M
Se crea una tabla intermedia con las PK de ambas entidades.
Formas normales: 1FN = valor único por columna · 2FN = en 1FN + dependencia de TODA la clave · 3FN = en 2FN + sin dependencias transitivas entre no-claves.

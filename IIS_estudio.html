Conceptos clave para las preguntas cortas (2,5 pts)

Ciclos de vida del software Muy preguntado
▼

Cascada

Fases secuenciales. Debes terminar cada fase antes de empezar la siguiente. Documentos entre fases.

Ciclo en V

Como cascada pero jerarquizado. Izquierda: descomposición. Derecha: integración.

Prototipo rápido

"Usar y tirar". Solo para ganar experiencia en análisis/diseño.

Prototipo evolutivo

Iterativo sobre cascada. Cada iteración da una versión más completa.

Espiral

4 cuadrantes: Planificación → Riesgo → Ingeniería → Evaluación. Guiado por riesgo.

XP (Extreme Programming)

4 valores: Sencillez, Comunicación, Retroalimentación, Valentía. Ciclos cortos.

Cascada vs V: el modelo en V añade jerarquía y diferencia verificación (¿cumple especificación?) de validación (¿cumple necesidades del usuario?).
Acoplamiento y cohesión Pregunta frecuente
▼

Objetivo: acoplamiento débil + cohesión alta

ACOPLAMIENTO (de peor a mejor) FUERTE Por Contenido → por Común → Externo MODERADO Por Control → por Etiqueta DÉBIL De Datos → Sin acoplamiento ← objetivo
COHESIÓN (de peor a mejor) BAJA Coincidental → Lógica → Temporal MEDIA De Comunicación → Secuencial ALTA Funcional → Abstraccional ← objetivo
Un módulo bueno hace una sola cosa bien definida (alta cohesión) y depende lo mínimo posible de otros módulos (acoplamiento débil).
Abstracción, herencia, genericidad Pregunta 2013
▼

Abstracción

Identificar elementos significativos ocultando los detalles irrelevantes. Tipos: funcional (funciones), tipo abstracto de datos (TAD), máquina abstracta.

Herencia

Los hijos heredan estructura y operaciones del padre y pueden ampliarlas. Permite reutilizar software. Ejemplo: Vehículo → Coche → Coche eléctrico.

Genericidad

Diseñar un elemento genérico válido para varios tipos. En C++: plantillas (templates). Ejemplo: Lista<T> funciona para cualquier tipo T.

Diferencia clave Abstracción vs Herencia: La abstracción define QUÉ hace un módulo ocultando el CÓMO. La herencia permite reutilizar e extender lo ya definido.
Pruebas de software Examen junio 2014
▼

Caja negra

Ignora estructura interna. Solo comprueba entrada-salida. Puede usarla el cliente. Métodos: clases de equivalencia, valores límite.

Caja transparente

Conoce estructura interna. Cubre todos los caminos. Métodos: cubrimiento lógico, pruebas de bucles.

Integración Big Bang

Todo de golpe. Solo válido para sistemas pequeños.

Descendente / Ascendente

Descendente: parte del módulo principal + sustitutos. Ascendente: parte de módulos hoja + gestores.

Una prueba tiene éxito si descubre un error. Si no aparecen errores, NO se garantiza la calidad: pueden existir errores que las pruebas no han cubierto.

UML

Diagramas de clases, estados y casos de uso

Estructura de una clase
┌─────────────────────┐ │ NombreClase │ ← mayúscula, sustantivo ├─────────────────────┤ │ -atributo: tipo │ ← - privado │ +atributo: tipo │ ← + público │ #atributo: tipo │ ← # protegido ├─────────────────────┤ │ +operacion(): tipo │ ← métodos └─────────────────────┘
Las 5 relaciones esenciales
——▷
Herencia: "B es un tipo de A". Flecha hueca. Ej: Coche hereda de Vehículo.
◆———
Composición: B no existe sin A. Rombo relleno en A. Ej: Edificio en Propiedad.
◇———
Agregación: A agrupa B, pero B puede existir solo. Rombo vacío en A. Ej: Propiedad en Grupo.
———
Asociación: A conoce o usa B. Línea continua con multiplicidad en ambos extremos.
- - ->
Dependencia: A usa B puntualmente. Línea discontinua con flecha.
Multiplicidades: 1 · 0..1 · 1..* · 0..* · N..M — ponlas SIEMPRE en los dos extremos de una asociación/composición/agregación.
Método para el examen (4 pasos)
Subraya sustantivos → candidatos a clase
Busca "es un tipo de" → herencia (▷)
Busca "tiene / contiene" → ¿puede existir sin el padre? No=◆, Sí=◇
Pon multiplicidades en todos los extremos
Ejercicio resuelto: Monopoly Examen sept. 2014
▼

40 casillas, 28 propiedades en grupos, hasta 6 jugadores, dinero, casas y hoteles.

Clases identificadas (sustantivos subrayados): Juego, Tablero, Casilla, Jugador, Propiedad, Grupo, Edificio Relaciones: Juego ◆──1──▶ 1 Tablero (composición) Juego ────1──▶ 2..6 Jugador (asociación) Tablero ◆──1──▶ 40 Casilla (composición) Casilla ──▷ Propiedad (herencia) Propiedad ◆──1──▶ 0..4 Edificio (composición) Propiedad ◇──1..*──▶ 1 Grupo (agregación) Jugador - - -> 0..* Propiedad (dependencia: posee)
¿Por qué Casilla → Propiedad es herencia? El enunciado dice "las casillas pueden ser propiedades". Propiedad es un tipo de Casilla.
¿Por qué Propiedad → Edificio es composición y no agregación? Los edificios no existen fuera de una propiedad. Si se borra la propiedad, desaparecen sus edificios.

Diagrama de estados UML Casi todos los exámenes
▼
Notación: ● ──▶ Estado inicial (círculo relleno) [Estado] ──evento──▶ Transición [Estado final] Punto de terminación: ◉ Ejemplo: puerta de garaje ● ──▶ [Cerrada] [Cerrada] ──botón──▶ [Abriendo] [Abriendo] ──completada──▶ [Abierta] [Abierta] ──botón──▶ [Cerrando] [Cerrando] ──completada──▶ [Cerrada] [Abriendo|Cerrando] ──botón──▶ [Parada] [Parada] ──botón──▶ [continúa en sentido contrario]
Truco: dibuja primero los estados (sustantivos de estado: Abierta, Cerrada…) y luego los eventos como flechas entre ellos.

Modelo E-R

Entidad-Relación + tablas relacionales

Símbolos del diagrama E-R
[ Entidad ] Rectángulo → clase de objeto ◇ Relación ◇ Rombo → verbo que une entidades (atributo) Elipse → propiedad de entidad/relación Min:Max Cardinalidad en cada extremo
Cardinalidades más comunes: 1:1 · 1:N · N:M (o 0:M, 1:N, 0:N...)
Traducción a tablas SQL
1 : 1
Se pueden fundir las dos entidades en una sola tabla.
1 : N
La clave primaria del lado "1" va como clave foránea en la tabla del lado "N".
N : M
Se crea una tabla intermedia con las claves primarias de ambas entidades.
Formas normales
1FN
Cada columna tiene un valor único (no colecciones ni listas).
2FN
En 1FN + todo atributo no clave depende de TODA la clave primaria.
3FN
En 2FN + cada columna no clave depende directamente de la clave primaria (no de otra no-clave).

DFD

Diagramas de flujo de datos

Símbolos del DFD
→ flecha Flujo de datos (con nombre encima) ( burbuja ) Proceso/transformación de datos [ Entidad ] Entidad externa (produce o consume) ═══════════ Almacén de datos
Jerarquía de niveles
Nivel 0 — Diagrama de contexto: el sistema completo como UNA sola burbuja con todas las entidades externas y flujos de entrada/salida.
Nivel 1 — DFD 0: explota la burbuja del nivel 0 en subprocesos. Los flujos deben coincidir exactamente con los del nivel 0.
Niveles 2, 3…: cada burbuja puede explotarse. Numeración: DFD 1.1, 1.2, 2.1…
Regla de balanceo: al explotar un proceso, los flujos que entran/salen del DFD resultante DEBEN ser exactamente los mismos que tenía el proceso antes de explotar.
